Object Oriented Programming Glossary

Terms & Definitions

Object: In object-oriented programming, an object can be considered an instance of a particular class. Furthermore, an object consists of a state and a behavior. A state can be considered the adjective-type attribute of an object. For example, a dog has a name(State 1) and a color(State 2). A behavior can be considered the verb-type attribute of an object. For example, a dog can bark(Behavior 1) and can fetch(Behavior 2). You can refer to the following link for further information: https://docs.oracle.com/javase/tutorial/java/concepts/object.html

Instantiation:Creation of an object. You can refer to the following link for further information: https://www.techopedia.com/definition/26857/instantiate

Class: A class is a blueprint for creating objects. They can provide initial values for a state (considered member variables or attributes). They can provide implementations of behaviors (considered member functions or methods). A class is a way of organizing information about a type of data so a programmer can reuse elements when making multiple instances of that data type. It is an example of encapsulation which will be discussed in further detail later. You can refer to the following link for further information: https://brilliant.org/wiki/classes-oop/

Namespace: Creates a name for objects of various kinds. It allows for objects to have unique names for proper identification. You can refer to the following link for further information: https://en.cppreference.com/w/cpp/language/namespace

Constructor: A method of a class or structure that initializes the creation of an object. The constructor is called automatically when an object is created. It can be used to se the values of the members of an object. You can refer to the following link for further information: https://www.techopedia.com/definition/5656/constructor

Fixture: There are multiple types of fixtures. You can refer to the following link for further information: https://stackoverflow.com/questions/12071344/what-are-fixtures-in-programming#:~:text=1.,the%20fixture%20of%20the%20test.
-- Test Fixture: A known state against which a test is running. It is a set or preconditions or the state needed to run a test.
-- Fixture: A file containing sample data.
-- Software Test Fixture: A process that sets up a required state.

Type Hint: Allows one to define the type of values that can be passed for each argument of a function or method. You can refer to the following link for further information: https://torquemag.io/2016/09/wordpress-developers-guide-type-hinting/

Type Cast: The process of converting the data type of the outcome of any operation to another data type. You can refer to the following link for further information: https://www.educba.com/what-is-type-casting/

Unit Test: Away of testing a unit which is the smallest piece of code that can be logically isolated in a system. You can refer to the following link for further information: https://smartbear.com/learn/automated-testing/what-is-unit-testing/

Static: Static is a storage class modifier. data members are shared by all instances of the class and are stored in one place. A static variable is defined when the storage class of static is made. You can refer to the following link for further information: https://docs.mql4.com/basis/oop/staticmembers

Instance Method: Method that belongs to an instance of a class and not the class itself. They belong to a particular object. You can refer to the following link for further information: http://www.cs.ecu.edu/karl/3200/fall15/Notes/Java/Class/method.html

Instance Property: A typical property declared in a class. You will have a copy of the property's value for each instance of the class. You can refer to the following link for further information: https://csharp.2000things.com/2011/03/03/259-static-vs-instance-properties/

Static Method: A method that belongs to a class rather than an instance of a class. It is part of the class definition. You can refer to the following link for further information: https://www.techopedia.com/definition/24034/static-method-java#:~:text=Java%208%20Interfaces-,What%20Does%20Static%20Method%20Mean%3F,that%20object%20of%20a%20class.

Static Property: Property that has a single value for the entire class. You can refer to the following link for further information: https://csharp.2000things.com/2011/03/03/259-static-vs-instance-properties/

Encapsulation: Occurs when each object keeps its state private and inside a class. Objects can manage their own state via methods. No other class can touch the state unless it is explicitly allowed. Public methods can be called by classes but they cannot modify the internal state. For example, you can feed a cat (public method called), but you cannot modify how hungry (state is hunger) a cat is. In essence, binding a private state and public state creates encapsulation. It prevents accidental modification of data. You can refer to the following link for further information: https://www.geeksforgeeks.org/encapsulation-in-python/

Inheritance: Allows a programmer to reuse common logic and extract the unique logic into a separate class. First, a parent class is create which is a class to easily hold all common functions and parameters that will be used in a child class. A child class is a class that inherits all of the attributes and functions assigned to the parent class. Each class adds only what is necessary for it while reusing common logic with the parent class. You can refer to the following link for further information: https://www.geeksforgeeks.org/inheritance-in-python/ The benefits of inheritance are:
-- Good real world representations
-- Provides reusability of the code
-- Transitive in nature

Polymorphism: Allows a programmer to use a collection of child classes and parent class methods. A coder can use a method that is implemented for a parent class inside a child class.A coder can use a class exactly like its parent so there is no confusion with mixing type. Each child class keeps its own methods as they are. This is done by defining a parent interface that can be reused. The parent interface has multiple common methods and each child class implements its own version of said common method. In other words, the same function name (but different signatures) can be used for different types. Polymorphism can be used in class methods, Inheritance, and with functions and objects. You can refer to the following link for further information: https://www.geeksforgeeks.org/polymorphism-in-python/

Abstraction: Helps the maintenance of large programs that have many separate objects in constant communication. Each object must only expose a high-level mechanism for using it. The mechanism must hide internal implementation details and only reveal operations relevant for other objects. For examples, a phone allows a user to click buttons for an output, but it does not demonstrate the process of retrieving the output to the user. In terms of coding, a public method should be accessed by any class. That class does not care or know how the method works. You can refer to the following link for further information: https://www.geeksforgeeks.org/abstract-classes-in-python/

SOLID: SOLID are 5 design principles that encourage maintainable, understandable, and flexible software. The 5 principle work well in conjunction with the 4 OOP fundamental concepts. You can refer to the following link for further information: https://www.baeldung.com/solid-principles
-- Single-Responsibility Principle: A class should only have on responsibility and only one reason to change. This allows for fewer test cases, fewer dependencies and well-organized classes.
-- Open-Closed Principle: A class should be open for extension but closed for modification. Doing so will prevent programmers from modifying existing code and creating new bugs.
-- Liskov Substitution Principle: Functions that use pointers or reference to base classes must be able to use objects of derived classes without knowing it. In other words, derived classes must be substitutable for their base class.
-- Interface Segregation Principle: Larger interfaces should be split into smaller ones. This ensures that anything a user interacts with belongs only that class.
-- Dependency Inversion Principle: Abstractions should not depend on details. Details should depend on abstraction. High-level modules should not depend on low-level modules. Both should depend on abstractions.

Design Patterns: Design Patterns are essentially pre-made blueprints that a programmer can customize to solve a design problem in code. They are concepts that can be implemented when designing a program with code. There are three categories of design patterns, Creational patterns, Structural patterns, and Behavioral patterns. You can refer to the following link for further information: https://refactoring.guru/design-patterns/catalog
-- Creational Patterns: Provide various object creation mechanisms. Good for increasing flexibility and reusing existing code. Examples include, factory method and abstract factory.
-- Structural Patterns: Explains how to assemble objects and classes into larger structures while keeping structures flexible and efficient. Examples include adapter and bridge.
-- Behavioral Patterns: Patterns concerned with algorithms and the assignment of responsibilities between objects. Examples include command and iterator.
